<objective>
Implement the TwiBot-native evaluation entry point so the newly trained native cascade can be
run on `test.json` and produce stable metrics/artifacts for v1.4 evidence and downstream comparison.

Purpose: satisfy `TRN-02` while reinforcing `TRN-03` through a separate native model-loading path.

Output:
- `evaluate_twibot20_native.py`
- Focused native evaluation tests
</objective>

<tasks>

<task>
  <name>Task 1: Build the TwiBot-native evaluation entry point</name>
  <files>evaluate_twibot20_native.py</files>
  <action>
    Create a standalone native evaluation script that:
    - loads the separate TwiBot-native joblib artifact by default
    - loads `test.json` and split-local edges through `twibot20_io`
    - scopes in the Phase 14 native extractors for inference
    - calls `predict_system()` and `evaluate_s3()`
    - writes stable JSON outputs for the native results and metrics payloads

    This script must not reuse the older Reddit zero-shot adapter behavior from `evaluate_twibot20.py`.
    The artifact schema should be explicit enough for Phase 16 comparison work to consume directly.
  </action>
  <verify>
    Prove the native evaluation path restores extractor overrides, returns the expected metric sections, and defaults to the native artifact path instead of `trained_system_v12.joblib`.
  </verify>
</task>

<task>
  <name>Task 2: Add focused native evaluation tests</name>
  <files>tests/test_evaluate_twibot20_native.py</files>
  <action>
    Add tests using small synthetic TwiBot JSON fixtures and stub systems. Cover:
    - default native model-path selection
    - native extractor override and restoration during inference
    - result/metrics JSON schema
    - overall/per-stage/routing metrics presence
    - separation from the legacy zero-shot evaluation path
  </action>
  <verify>
    Tests should assert output keys and at least a few behavioral expectations rather than only checking that files exist.
  </verify>
</task>

</tasks>
